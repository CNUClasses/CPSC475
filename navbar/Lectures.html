Builder Pattern Demo Do not let users arbitrairly construct a complex object using variable length constructors, or getters and setters. These methods either do not scale or do not gurantee a properly constructed object. Instead make the objects constructor private and add a static class builder to it. The builder is responsible for collecting and verifing all fields of the object before the object is constructed. Once verified, the builder builds and returns an instance of the object through its build() method.

Long running task on the UI thread cause a non-responsive UI and a probable 'Application not responding (ANR) error.
Therefore all long running tasks should take place on a separate thread. See below projects for ANR demo

Threads that are non-static inner classes of an activity hold an implicit reference to their enclosing activity which prevents the activity from being garbage collected when the device is rotated and the activity is torn down and recreated.
Fix this problem by making the threads static inner classes

A larger problem is what happens to a thread when the device is rotated? If a thread trys to use a reference to a widget in the activity that is being destroyed it triggers a null pointer exception.
Fix this by attaching and detaching a parent activity to a thread when needed. But we are back to holding a reference to the activity and some very fussy code to get right. Plus there is tight coupling between the thread and the activity (We will fix this problem soon by using something called Mutable live Data which is an Android implementation of the Observer Design pattern.)

ViewModel Overview
LiveData overview How Android uses the observer software design pattern to host all relevant data in the ViewModel, automatically update the Activity when the data in the ViewModel changes, and decouple the ViewModel from the Activity. Using this method you don't have to worry about leaking the activity when its destroyed since threads no longer hold a reference to it. The android framework manages all state adjustments associated with Activity configuration changes.
This operates much like a preference change listener which is notified when a change is made to a preference in a preference activity.
Observer design pattern Decouples components, a subject manages data and notifies observers when that data changes, the heart of LiveView data
 In Class Lab -Demo using ViewModel, LiveData and Java Thread - the reccomended Google way
 Have the Activity listen for changes in liveData objects in the ViewModel. These objects correspond to views in the Activity, but the Java Thread and the ViewModel have no reference to the Activity. The Activity is updated by Observers listening for changes in the ViewModel. No chance of leaking the activity. No coupling between the Activity and the thread.
